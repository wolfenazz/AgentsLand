# Plan: Add Stepper Mode Toggle to Setup Page

## Summary
Add a settings option to toggle the setup page between the current single-page layout and a new multi-step stepper layout. The stepper will be a high-end, professional wizard-style UI with animated transitions, step validation, and a polished visual design.

## Current Architecture
- `SetupScreen.tsx` renders a single scrolling page with: logo, warnings, `WorkspaceConfigForm` (template, workspace details, project init, layout, agent fleet), and `CliToolsTable`
- `WorkspaceConfigForm.tsx` contains all form sections stacked vertically
- Settings are stored in `appStore.ts` via Zustand with `persist` middleware
- Settings UI follows a consistent pattern with `SettingsAppearance.tsx` etc.

## Changes

### 1. Add `setupViewMode` to `appStore.ts`

**State additions:**
- `setupViewMode: "page" | "stepper"` — defaults to `"page"`
- `setSetupViewMode: (mode: "page" | "stepper") => void`

**Persist:** Add `setupViewMode` to the `partialize` config so it's saved across sessions.

### 2. Add Toggle in `SettingsAppearance.tsx`

Add a new setting row under the existing "Preferences" section:

```
Setup Layout
Choose between single-page or multi-step wizard for workspace setup

[ Page  |  Stepper ]  ← pill-toggle buttons
```

Uses the same styling pattern as existing toggle switches and buttons.

### 3. Create `SetupStepper.tsx`

New component at `src/components/setup/SetupStepper.tsx`.

**Steps definition (5 steps):**

| # | Step | Icon | Content | Validation |
|---|------|------|---------|------------|
| 1 | Template | grid icon | `WorkspaceTemplatePicker` | Always valid (template is optional) |
| 2 | Workspace | folder icon | Workspace Name input + `DirectorySelector` + `InitializeWorkspace` | `selectedPath` must be set; `workspaceName` if not external mode |
| 3 | Layout | layout-grid icon | `LayoutSelector` + `IdesSelector` | Always valid |
| 4 | Agents | cpu/robot icon | `AgentFleetConfig` + allocation validation | `isAllocationValid` must be true |
| 5 | Launch | rocket/play icon | Summary card (path, name, layout, agent counts) + Execute button + Cancel | `isValid` must be true |

**Template-aware smart flow:**

When the user selects a template (any non-"custom" template), the stepper simplifies:

- **Steps 3 & 4 auto-complete** — the template pre-fills layout + agent allocation, so those steps show a brief "Configured by template" summary and auto-advance (or allow the user to click to customize).
- The user effectively goes: **Template → Workspace → Launch** (3 clicks).
- The skipped steps (Layout, Agents) still appear in the stepper bar but are marked with a template icon/badge to show they're auto-configured.
- Clicking a skipped step still opens it for manual override.
- If the user switches back to "Custom", all steps become required again.

Implementation: Each step has a `isAutoCompleted` flag derived from `selectedTemplateId !== 'custom'`. When `isAutoCompleted` is true:
  - The step renders a compact summary card instead of the full form
  - The Next button auto-activates
  - The step indicator shows a special "auto" state (small template badge)
  - The user can still click the step to expand and customize

**Props:** Same data props that `WorkspaceConfigForm` receives, plus `onCreateWorkspace`, `onCancel`, `isLaunching`, error states.

**Layout design:**
```
┌─────────────────────────────────────────────────┐
│  ← TopBar (same as current)                   → │
├─────────────────────────────────────────────────┤
│  ○─────○─────○─────○─────○                      │
│  01    02    03    04    05                      │
│  Temp  Work  Layo  Agen  Laun                   │
│  late  space  out   ts    ch                    │
├─────────────────────────────────────────────────┤
│                                                 │
│            [ Step Content Area ]                │
│         (scrollable, animated transitions)       │

```

**Stepper header design details:**
- Horizontal stepper bar at the top of the content area
- Each step has: number circle (filled when active, checkmark when complete, outlined when pending), label text, connecting line
- Active step: white/bright circle with glow, bold label
- Completed steps: emerald checkmark, muted connecting line
- Future steps: dimmed, outlined circles
- Smooth color transitions on step changes

**Step content area:**
- Only renders the current step's content
- Smooth fade/slide transition using Framer Motion (already a dependency)
- `CliToolsTable` shown in a collapsible section at the bottom of every step (or just on the Launch step)

**Footer actions:**
- Left side: Cancel button (if `hasOpenWorkspaces`)
- Right side: Back button (disabled on step 1), Next button (disabled if current step validation fails), Execute button (on final step, same styling as current Execute button)
- Status indicator showing configured slots count

### 4. Modify `SetupScreen.tsx`

- Import `useAppStore` to read `setupViewMode`
- Import the new `SetupStepper` component
- In the `<main>` section, conditionally render:
  - If `setupViewMode === "stepper"` → render `SetupStepper` with all required props
  - Else → render existing page layout (current code, unchanged)

The stepper gets the same data flow — all the workspace config state, handlers, and callbacks come from `useWorkspace()` hook just like the current form does. No logic duplication.

## Key Design Decisions

1. **No shared state refactoring** — The stepper uses the same `useWorkspace()` hook data. It's purely a UI layout change, not a logic change.
2. **Step validation is local** — Each step defines its own `canAdvance()` check. The Next button is disabled when the check fails, and a subtle inline message explains why.
3. **Template-aware simplification** — When a non-custom template is selected, Layout and Agent steps auto-complete with a summary view. The user can still click to customize. Custom mode requires all steps.
4. **CliToolsTable placement** — Shown as a collapsible section at the bottom of every stepper step so it's always accessible.
5. **Animations via Framer Motion** — Already a project dependency. Step transitions use `AnimatePresence` with slide + fade.
6. **The existing page mode is untouched** — All current code remains identical; the stepper is an additive feature.
