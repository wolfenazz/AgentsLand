# Plan: Node.js First-Time Installation Check Gate

## Summary

Add a Node.js detection gate that blocks the app on first launch if Node.js is not installed. The user sees a full-screen prompt with an "Install Node.js" button (opens `https://nodejs.org/en/download/current`), then an "I've installed it" button that re-checks. If still missing, a warning message is shown explaining that CLI agents won't work. Once Node.js is detected (or was already installed), the app proceeds to the normal setup screen.

## Architecture

The existing `check_prerequisites` Rust command already detects Node.js with version checking. We'll leverage it and add a new lightweight Rust command `check_nodejs` for fast single-check, plus a new frontend component `NodeJsCheckGate` that wraps the app initialization.

### 2. Frontend — New `NodeJsCheckGate` component

**New file:** `app/src/components/setup/NodeJsCheckGate.tsx`

A full-screen overlay component with three states:

| State | UI |
|-------|----|
| `checking` | Loading spinner + "Checking system requirements..." |
| `not-installed` | Node.js icon/logo, warning message about CLI agents needing Node.js, "Install Node.js" button (opens browser to `https://nodejs.org/en/download/current` via `invoke('open_url')`), and "I've installed it" button that re-runs `check_nodejs` |
| `install-failed` | Same as `not-installed` but with an additional error banner: "Node.js is still not detected. CLI agents (Claude, Codex, Gemini, OpenCode, Kilo) will not work properly. Please restart the app after installing Node.js." with a "Continue anyway" link at the bottom |
| `installed` | Renders children (the rest of the app) |

Flow:
1. On mount, call `invoke<PrerequisiteStatus>('check_nodejs')`.
2. If `installed && meetsMinimum` → state = `installed`, render children.
3. If not installed → state = `not-installed`.
4. "Install Node.js" button → `invoke('open_url', { url: 'https://nodejs.org/en/download/current' })`.
5. "I've installed it" button → re-run `check_nodejs` with a loading spinner.
6. If still not installed → state = `install-failed` with warning message.
7. If now installed → state = `installed`.
8. "Continue anyway" link → skip gate (state = `installed`), store a flag so gate isn't shown again this session.

The component also stores `nodeJsCheckDismissed` in a React ref (session-only, NOT persisted) so if the user clicks "Continue anyway", they aren't blocked again during the same session.

### 5. Handle workspace auto-restore

**File:** `app/src/App.tsx`

The current workspace auto-restore logic runs on mount. It needs to be conditional — only restore workspace if the Node.js check passes. Move the auto-restore logic into a callback that fires after the gate passes.

In the `useEffect([], ...)` that handles auto-restore, check if `view === 'nodejs-check'` first. If it is, skip auto-restore. The auto-restore will happen when the view changes to `'setup'` or `'workspace'`.

Actually, since the default view is now `'nodejs-check'`, the auto-restore in the mount effect won't trigger workspace view change. We need to run the auto-restore when transitioning from `'nodejs-check'` to the appropriate view. The `NodeJsCheckScreen` component will handle this:
- If Node.js is installed AND there are saved workspaces to restore → call a new callback `onReady(workspaceId)` that switches to workspace view directly.
- Otherwise → call `onContinue()` to go to setup.

**Revised approach for `NodeJsCheckScreen` props:**

```typescript
interface NodeJsCheckScreenProps {
  onReady: () => void; // Called when Node.js is confirmed, handles auto-restore logic
}
```

The `onReady` callback in App.tsx will contain the workspace auto-restore logic:

```typescript
const handleNodeJsReady = () => {
  if (saveWorkspaceState && openWorkspaces.length > 0) {
    const targetId = activeWorkspaceId || openWorkspaces[0]?.id;
    if (targetId) {
      switchWorkspace(targetId);
      setView('workspace');
      return;
    }
  }
  setView('setup');
};
```

And the existing auto-restore in the mount `useEffect` should be removed (moved to `handleNodeJsReady`).

## Check Frequency: First Launch Only

The check runs once. A persisted boolean `nodejsCheckPassed` is stored in the Zustand store. Once the gate is passed (Node.js found or user skipped), it's set to `true` and the gate never appears again.

On subsequent launches, the mount effect in App.tsx checks `nodejsCheckPassed`:
- If `true` → skip the gate entirely, go straight to setup or workspace restore
- If `false`/undefined (first launch) → show the gate

### Store addition (`app/src/stores/appStore.ts`)

Add to state interface and initial state:
```typescript
nodejsCheckPassed: boolean;
setNodeJsCheckPassed: (passed: boolean) => void;
```

Add to `partialize` so it's persisted:
```typescript
nodejsCheckPassed: state.nodejsCheckPassed,
```

### App.tsx mount logic (revised)

```typescript
useEffect(() => {
  initWindowPlatform().then(setIsWindows);

  if (nodejsCheckPassed) {
    // Already passed before — restore workspace or go to setup
    if (saveWorkspaceState && openWorkspaces.length > 0) {
      const targetId = activeWorkspaceId || openWorkspaces[0]?.id;
      if (targetId) {
        switchWorkspace(targetId);
        setView('workspace');
      }
    } else {
      setView('setup');
    }
  }
  // If nodejsCheckPassed is false, view stays "nodejs-check"
}, []);
```
